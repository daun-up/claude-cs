문제 01 오답 포인트: 변수는 참조타입, 메서드는 실제 인스턴스 타입 기준
Java
A a = new B();
// 메서드 → 실제 인스턴스(B) 기준
a.method(); // → B의 method() 실행
// 필드 → 참조타입(A) 기준
a.x;  // → A의 x 값
// B에만 있는 멤버 접근 불가
// a.onlyInB(); ← 컴파일 에러
규칙 1 메서드 호출 → 런타임 실제 인스턴스 타입 (동적 바인딩)
규칙 2 필드·접근 범위 → 컴파일타임 참조타입 기준
규칙 3 static 메서드 → 참조타입 기준 (동적 바인딩 X)

예제 1 — 오버라이딩 + 필드 숨김 문제 01 오답
SSAFY 오답 유형 코드
class A {
    int x = 10;
    void method() { System.out.println("A: " + x); }
}
class B extends A {
    int x = 20;
    @Override
    void method() { System.out.println("B: " + x); }
}

A a = new B();
System.out.println(a.x);  // ← ?
a.method();               // ← ?
🎯 출력 결과 예측 → 클릭해서 확인
10 ← 필드는 참조타입(A) 기준 → A의 x = 10 B: 20 ← 메서드는 실제 인스턴스(B) 기준 → B의 method() → B의 x = 20 오답 이유: "참조타입이 A니까 A의 method() 호출" 이라는 오해 → 메서드는 항상 런타임 객체 타입 기준 (동적 바인딩)

예제 2 — 참조값 전달 원리 문제 22 오답 (28%)
최저 정답률 문제 — Java는 항상 값 전달(pass-by-value)
SSAFY 오답 유형 코드
class Data { int x; }

static void change(Data data) {
    data.x = 1000;    // 원본 객체 내부 수정 → 반영 O
    data = new Data(); // 로컬 참조변수 재할당 → 반영 X
    data.x = 2000;    // 새 객체 수정 → 원본과 무관
}

Data d = new Data();
d.x = 5;
change(d);
System.out.println(d.x); // ← ?
🎯 출력 결과 예측 → 클릭해서 확인
1000 ① data.x = 1000 → 참조값(주소)으로 원본 객체 내부 수정 → 반영 O ② data = new Data() → 로컬 복사본(참조값)만 변경 → 원본에 영향 없음 ③ data.x = 2000 → 새 객체에만 씀 → 원본 d와 무관 핵심: Java는 참조타입도 참조값(주소의 복사본)을 전달 내부 상태 변경 O ≠ 참조 자체 변경 X
흔한 오해 "객체를 전달하면 메서드 내에서 참조 자체를 바꿀 수 있다" → 틀림! 재할당(data = new Data())은 외부에 영향 없음

static int test() {
    try {
        return 1;   // 먼저 실행
    } finally {
        return 3;   // 이 값이 반환됨!
    }

System.exit() → finally 미실행 (문제 03)
try {
    System.exit(0);  // JVM 강제 종료
} finally {
    // 절대 실행 안 됨!
    System.out.println("실행 안 됨");
}
상황finally 실행?
try 정상 종료 / 예외 발생✅ 실행
try에서 return / continue / break✅ 실행 (finally 먼저)
System.exit()❌ 미실행 (JVM 종료)
JVM 크래시 / 무한 루프❌ 미실행
핵심 finally의 return문은 try/catch의 return을 덮어씀 — 실무에서 finally에 return 쓰는 것은 안티패턴!

List<String> list = Arrays.asList("java", "spring", "service");

2차원 배열 clone — 얕은 복사 문제 04 오답 (39%) 문제 21 오답
SSAFY 오답 유형 코드
int[][] arr1 = { {1, 2}, {3, 4} };
int[][] arr2 = arr1.clone();  // 얕은 복사!

arr2[0][0] = 10;           // 내부 배열 공유 → arr1[0][0]도 10
arr2[1] = new int[]{3};   // arr2[1] 참조만 교체 → arr1[1] 영향 없음

System.out.println(arr2[1][0]); // 3   ← 새 배열

// 깊은 복사 방법
int[][] deep = Arrays.stream(arr1).map(int[]::clone).toArray(int[][]::new);

포스트잇 — 오개념 교정
18개 오답에서 추출한 핵심 오개념 — 틀린 생각(취소선) vs 올바른 개념
#01
📌 다형성 — 메서드 호출
참조타입이 부모면 부모 메서드가 호출된다
메서드는 실제 인스턴스(런타임) 타입 기준 → 오버라이딩된 자식 메서드 실행 (동적 바인딩)
#01
📌 필드 vs 메서드 바인딩
필드도 오버라이딩되므로 자식 필드가 사용된다
필드는 오버라이딩 없음 → 참조타입(컴파일타임) 기준
부모 참조변수면 부모 필드
#22
📌 Pass by Value
객체 전달 시 참조 자체가 전달되므로 재할당하면 반영된다
참조값(주소 복사본) 전달
재할당 → 반영 안 됨
내부 필드 수정 → 반영 됨
#03
📌 finally — System.exit()
finally는 무조건 항상 실행된다
System.exit() → JVM 강제 종료
→ finally 미실행!
일반 return/break → 실행 O
#18
📌 finally return 덮어쓰기
try의 return이 먼저 선언됐으니 그 값이 반환된다
finally 블록의 return이
try/catch의 return을 덮어씀
→ finally return 값이 최종 반환
#04
📌 2D 배열 clone
clone()은 깊은 복사이므로 내부 배열도 독립 복사된다
clone()은 얕은 복사
→ 1차원 배열 참조만 복사
내부 배열은 공유됨!
#08
📌 인터페이스 접근제한자
인터페이스에서 protected 메서드를 선언할 수 있다
인터페이스 추상/default/static
→ public만 허용
protected 선언 → 컴파일 에러!
#15
📌 Interface default 우선순위
상위 인터페이스의 default가 우선된다
구현 클래스 > 하위 인터페이스 > 상위 인터페이스
충돌 시 구현 클래스에서 직접 오버라이딩 필요
#11
📌 Generic 불공변
String이 Object 자식이므로 Box<String>을 Box<Object>에 대입 가능
제네릭은 불공변(invariant)
Box<Object> ≠ Box<String>
와일드카드 Box<?> 사용 필요
#12
📌 final 인스턴스 변수
final 변수는 반드시 선언과 동시에 초기화해야 한다
final 인스턴스 변수는
선언 시 OR 생성자에서 초기화 가능
둘 중 하나 반드시 해야 함
#26
📌 생성자 호출 순서
자식 생성자가 먼저 실행된다
항상 최상위 부모 먼저!
Object → 부모 → 자식
자식 생성자 호출 시마다 부모 우선
#06
📌 static 메서드 제약
static 메서드에서 인스턴스 변수를 직접 사용할 수 있다
static 메서드 → 인스턴스 변수 접근 불가
this 사용 불가
오버라이딩 불가 (숨김만)
#28
📌 문자열 연산 순서
10 + 20 + "Java" + 10 + 20 = "Java"가 붙은 뒤 계산
왼쪽→오른쪽 순차 평가
10+20=30 → "30Java" → "30Java1020"
문자열 만나기 전까지는 정수 덧셈
#07
📌 default 접근제한자
default는 자식 클래스면 패키지 달라도 접근 가능
default = package-private
다른 패키지 자식 클래스도 접근 불가!
자식 접근 → protected 이상 필요
#10
📌 함수형 인터페이스
@FunctionalInterface는 추상 메서드 개수 제한이 없다
추상 메서드 반드시 1개
default/static은 여러 개 허용
0개 or 2개+ 추상 메서드 → 컴파일 에러
#29
📌 continue vs break
continue는 반복문을 완전히 종료한다
continue → 현재 반복만 건너뜀
(다음 반복 계속 수행)
break → 반복문 완전 종료
